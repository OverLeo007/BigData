## 1 Цель

Цель работы состоит в описании проекта Big Data-аналитики для сбора, хранения, обработки и анализа данных о качестве
атмосферного воздуха в Красноярске.

В качестве проекта выбрана система **KrasAir Analytics**. Она предназначена для работы с данными мониторинга воздуха,
метеорологическими показателями и сведениями о периодах неблагоприятных метеорологических условий. Основной объект
анализа, временные ряды показателей загрязнения воздуха по площадкам мониторинга города.

## 2 Задачи

Для достижения цели необходимо описать предполагаемые:

- а) источники данных, из каких хранилищ они должны "подгружаться", в том числе какие технологии хранения использованы;
- б) методы и средства сбора, предобработки (при необходимости) и отображения данных;
- в) технологии, методы и средства обработки данных;
- г) технологии и методы анализа данных, итоговые документы;
- д)реестр заинтересованных лиц и способы проведения их информирования, собеседования и интервьюирования о возможностях
  методов и инструментов больших данных.

## 3 Описание проекта

### 3.1 Общая характеристика проекта

Проект **KrasAir Analytics** представляет собой аналитическую платформу для изучения качества воздуха в Красноярске.
Система должна собирать данные из открытых источников, приводить их к единому формату, хранить историю измерений и
строить аналитические представления для разных групп пользователей.

Основные показатели для проекта – PM2.5, PM10 и AQI. Эти данные удобно анализировать как временные ряды: можно смотреть
значения по часам, дням, неделям, станциям мониторинга и периодам НМУ (Неблагоприятных Метеорологических Условий).
Дополнительно учитываются температура, влажность, давление, скорость ветра, осадки и календарные признаки.

Данные мониторинга можно использовать для понятной аналитической задачи: посмотреть, когда и где загрязнение растет, как
оно связано с погодой и отличаются ли периоды НМУ от обычных дней.

### 3.2 Источники данных

Основным источником данных выбран сайт мониторинга состояния атмосферного воздуха в
Красноярске [air.krasn.ru](https://air.krasn.ru). В справочном разделе сайта указано, что раздел «Данные» содержит
подразделы для просмотра, Excel-выгрузки и API. API реализован как REST-сервис, а ответы возвращаются в формате JSON.
Там же указано, что Excel-выгрузка предназначена для загрузки сырых данных с выбором периода, площадок мониторинга и
интервала агрегации.

В системе `air.krasn.ru` представлены два блока информации: метеорологические характеристики и данные по загрязнению
воздуха. К метеорологическим характеристикам относятся температура, давление и относительная влажность, а к данным
загрязнения, концентрации взвешенных частиц PM10 и PM2.5. На основе этих измерений рассчитывается Instant AQI.

Второй источник, сведения о НМУ.
На [сайте администрации Красноярска](https://www.admkrsk.ru/citytoday/ecology/Pages/NMU.aspx) НМУ описываются как
краткосрочное сочетание метеорологических факторов, при котором загрязняющие вещества накапливаются в приземном слое
воздуха. На этой же странице публикуются предупреждения с датами и временем действия.

Для дополнительного погодного слоя может использоваться Open-Meteo Historical Weather API. Сервис предоставляет
исторические погодные данные по координатам, включая температуру, влажность, осадки, давление, скорость и направление
ветра. В документации указано, что доступны исторические данные с 1940 года, а набор переменных можно запрашивать через
endpoint `/v1/archive`.

### 3.3 Хранилища данных и логическая структура хранения

В проекте предполагается использовать несколько слоев хранения. Такое разделение нужно, чтобы не смешивать исходные
файлы, очищенные данные и готовые витрины для отчетов.

**Raw layer** хранит данные в исходном виде. В этот слой попадают JSON-ответы API, Excel-выгрузки, HTML-страницы с
сообщениями о НМУ и погодные JSON-файлы. Эти данные не исправляются вручную. Если позже изменятся правила очистки,
обработку можно повторить с исходных файлов.

**Staging layer** хранит данные после первичной нормализации. На этом уровне уже приведены типы полей, даты, названия
станций и единицы измерения. Данные предлагается хранить в формате Parquet.

**Data marts layer** содержит витрины для аналитики. Эти таблицы уже рассчитаны под конкретные вопросы: средние значения
по станциям, статистика по дням, сравнение периодов НМУ и обычных периодов, рейтинг станций по уровню загрязнения.

### 3.4 Сбор данных

Сбор данных строится как регулярный процесс. Для каждого источника задается свой загрузчик.

Данные с `air.krasn.ru` могут загружаться двумя способами. Основной способ, REST API с JSON-ответами. Резервный способ,
Excel-выгрузка с сайта.

Для НМУ сбор сложнее, потому что исходные сообщения публикуются в текстовом виде. Их можно сохранить как HTML или текст,
а затем извлекать даты и интервалы действия НМУ регулярными выражениями. Если автоматический парсинг окажется
нестабильным предполагается вручную подготовить csv.

Погодные данные из Open-Meteo загружаются по координатам Красноярска. В запросе указывается период и набор переменных:
скорость ветра, направление ветра, осадки, температура, влажность и давление.

### 3.5 Предобработка данных

Предобработка нужна, потому что источники отличаются по структуре и качеству. Основная задача, привести все данные к
единой временной шкале и получить таблицу, с которой можно строить отчеты.

Для данных мониторинга воздуха выполняются следующие операции:

- приведение даты и времени к единому формату;
- проверка отрицательных значений PM2.5, PM10 и AQI;
- обработка пропусков без автоматического удаления записей;
- добавление флага качества данных;
- разметка резких скачков как возможных выбросов.

Для данных о НМУ выполняется нормализация текста. Из сообщения извлекаются дата публикации, начало периода, окончание
периода и режим. После этого период НМУ разворачивается до часового уровня, чтобы его можно было соединить с измерениями
воздуха.

Для отображения данных предлагается использовать Apache Superset или Apache Zeppelin. Оба инструмента подходят для
учебного проекта: Superset удобен для дашбордов, Zeppelin удобен для ноутбуков с кодом, SQL-запросами и графиками.

Таблица 1 – Представления системы

|--------------------------|--------------------------------------------------------------|
| Общая сводка             | текущие и средние значения PM2.5, PM10, AQI                  |

можно показать средний PM2.5 за сутки, максимальный AQI за день, число часов с повышенным загрязнением и станции, где
значения чаще всего были выше выбранного порога.

### 3.7 Технологии и средства обработки данных

В пакетной обработке основным инструментом является Apache Spark. Он подходит для объединения файлов из разных
источников, обработки временных рядов, группировок по станциям и периодам, расчета агрегатов и формирования витрин.

Для потоковой части может использоваться Apache Kafka. В промышленном сценарии в Kafka поступали бы новые измерения с
датчиков или сервисов мониторинга. В учебном прототипе поток можно имитировать: загрузчик читает исторические записи из
файла и отправляет их в Kafka как последовательность событий. Это позволяет проверить преобразование, фильтрацию и
агрегацию данных без доступа к реальному потоку датчиков.

Для хранения витрин можно использовать PostgreSQL. Если объем данных станет большим и потребуется быстро строить отчеты
по временным интервалам, лучше подойдет ClickHouse. Для файлового хранения очищенных данных выбран Parquet, потому что
этот формат хорошо работает со Spark и аналитическими запросами.

### 3.8 Методы анализа данных

В проекте предполагаются несколько уровней анализа.

Первый уровень, описательная аналитика. Она отвечает на простые, но важные вопросы: какие значения PM2.5 и PM10
наблюдались по станциям, как менялся AQI по дням, в какие часы чаще фиксировались высокие значения.

Второй уровень, сравнительный анализ. Здесь сопоставляются разные периоды и условия: дни с НМУ и без НМУ, зимний и
летний периоды, будние и выходные дни, площадки мониторинга в разных частях города.

Третий уровень, анализ связи загрязнения с метеофакторами. Для этого используются температура, влажность, давление,
ветер и осадки. Например, можно проверить, как средний PM2.5 меняется при слабом ветре и при более сильном ветре. Такой
анализ не доказывает источник загрязнения, но помогает найти устойчивые закономерности.

Четвертый уровень, анализ аномалий. Аномалией считается значение, которое резко отличается от обычного уровня для этой
станции и этого времени. В проекте можно использовать простые правила: межквартильный размах, сравнение с медианой за
тот же час суток или пороговые значения для выбранного показателя.

### 3.11 Информирование, собеседования и интервьюирование заинтересованных лиц

Информирование заинтересованных лиц нужно проводить до полноценного внедрения системы. Иначе можно сделать технически
правильный дашборд, которым неудобно пользоваться.

Для жителей важны простые объяснения. Им не обязательно видеть все технические поля. Нужны понятные графики, расшифровка
PM2.5, PM10 и AQI, а также сравнение текущего состояния с обычным уровнем.

Для администрации и экологических служб важнее агрегированные отчеты. Им нужны периоды, станции, динамика, превышения,
сравнение НМУ и обычных дней. Также важна выгрузка данных в привычных форматах, например Excel или PDF.

Основные способы информирования:

- презентация проекта с описанием источников, архитектуры и результатов;
- демонстрация дашборда на небольшом историческом наборе данных;
- пояснительная записка с описанием показателей и ограничений;
- регулярная аналитическая сводка за неделю или месяц;
- техническое описание таблиц, полей и правил загрузки данных.

Собеседования можно проводить в формате коротких интервью. Для администрации и экологических служб стоит задать вопросы
о нужной детализации, периодичности отчетов и форматах выгрузки. Для жителей, о понятности показателей и удобстве
визуализации.

По итогам интервью требования к системе уточняются.
